Добрый!

Сформировали критерии для того, чтобы собрать входные данные и проверить стабильность работы модели на более широком диапазоне

Общие требования ко всем наборам
•	Формат данных: только текст (без изображений, схем, диаграмм). Допускается форматирование (списки, заголовки, моноширинный шрифт для кода/параметров), но без внешних медиа.
 
•	Объём: 300–800 слов на набор (вариативно в зависимости от плотности информации).
 
•	Язык: русский (допускается включение технических терминов на английском, например, названия API методов, статусов HTTP).
 
•	Запрещено: добавлять явные подсказки для генерации тест кейсов (например, «на основе этого напиши 5 тест кейсов»). Входные данные должны выглядеть как реальная документация, а не как задание для ассистента.
 
---------------------------------------------------------------------------------------------------------------------------------------------------------------------
 
Критерии к каждому набору
Набор 1: структурированная и полная техническая документация (базовый эталон)
Цель: задать «золотой стандарт» — чёткая, полная, хорошо структурированная информация для сравнения с остальными наборами.
 
Содержание:
 
•	Краткое описание функционала (1–2 абзаца).
 
•	Детальное описание API эндпоинтов: URL, метод, параметры (обязательные/опциональные), примеры запросов/ответов, коды ошибок.
 
•	Бизнес правила и ограничения (валидация, лимиты, условия).
 
•	Сценарии использования (2–3 примера).
 
•	Ожидаемое поведение при граничных условиях.
 
•	Особенности: высокая полнота, чёткая структура, минимум двусмысленности.
 
Пример структуры:
 
Сервис бронирования отелей.
Эндпоинт: /api/v1/hotels (GET) — поиск отелей.
Параметры: location (обязательный), checkIn, checkOut, guests…
Бизнес правило: даты должны быть в будущем, checkOut > checkIn.
Ошибки: 400 при неверных датах, 404 если отелей нет…
 
---------------------------------------------------------------------------------------------------------------------------------------------------------------------
 
Набор 2: фрагментированная и неполная документация (проверка на устойчивость к недостатку данных)
Цель: проверить, как ассистент генерирует тест кейсы при нехватке информации, и умеет ли корректно обозначать пробелы.
 
Содержание:
 
•	Обрывочные описания API без полной спецификации (есть URL и метод, но нет параметров или примеров).
 
•	Частичные бизнес правила (указано «валидация дат», но не сказано, какие именно ограничения).
 
•	Упоминания о функционале без деталей («система отправляет уведомления», но не уточняется, когда и какие).
 
•	Пропущенные граничные условия и обработка ошибок.
 
•	Особенности: намеренно низкая полнота, много «белых пятен», но есть ключевые сущности (эндпоинты, основные поля).
 
Пример:
 
/api/v1/hotels — поиск отелей. Метод GET.
Нужно передавать даты.
Если отелей нет — вернуть пустой список.
Есть фильтрация, но не сказано по каким полям.
 
---------------------------------------------------------------------------------------------------------------------------------------------------------------------
 
Набор 3: бизнес ориентированная документация с пользовательскими историями (проверка понимания бизнес контекста)
Цель: оценить, насколько ассистент может извлекать тестовые сценарии из нетехнического описания, ориентированного на пользователя и бизнес.
 
Содержание:
 
•	Пользовательские истории в формате «Как [роль], я хочу [действие], чтобы [результат]».
 
•	Описание бизнес целей и KPI (например, «сократить время бронирования до 2 минут»).
 
•	Приоритеты и важность разных сценариев.
 
•	Ограничения со стороны бизнеса (например, «для корпоративных клиентов — особые условия»).
 
•	Неформальные описания процессов («пользователь выбирает отель, сравнивает цены, бронирует…»).
 
•	Особенности: минимум технических деталей, фокус на целях и ожиданиях пользователя, возможны неоднозначности в трактовке требований.
 
Пример:
 
Как гость, я хочу видеть только доступные отели на выбранные даты, чтобы не тратить время на недоступные варианты.
Для корпоративных клиентов цены должны быть ниже, и это должно отображаться сразу при поиске.
Важно, чтобы процесс бронирования занимал не более 2 минут.
 
---------------------------------------------------------------------------------------------------------------------------------------------------------------------

Набор 4: смешанная документация с противоречиями и шумом (проверка на обработку конфликтов и фильтрацию шума)
Цель: протестировать устойчивость к противоречивой и зашумлённой информации: дубли, конфликты требований, нерелевантные заметки.
 
Содержание:
 
•	Комбинация фрагментов из технической документации, заметок разработчиков, черновиков требований и случайных комментариев.
 
•	Противоречивые утверждения (например, «даты обязательны» vs «можно искать без дат»).
 
•	Дублирующиеся описания с небольшими отличиями.
 
•	Нерелевантные или устаревшие заметки («раньше было так…», «планируем в следующей версии…»).
 
•	Неформатированный текст с опечатками и стилистическими ошибками.
 
•	Особенности: низкая согласованность, наличие явных и скрытых конфликтов, шум, который нужно отфильтровать.
 
Пример:
 
Поиск отелей без дат тоже работает.
Даты обязательны для поиска.
Раньше искали без дат, теперь только с датами.
Эндпоинт /api/v1/hotels, метод GET.
Примечание: это черновик, не использовать в проде.
Фильтрация по цене есть. Фильтрации нет.

Набор 5: высокоуровневая и абстрактная документация (проверка способности к декомпозиции и домысливанию в рамках разумного)
Цель: оценить, как ассистент работает с очень общими, абстрактными формулировками без конкретных деталей.
 
Содержание:
 
•	Общее описание продукта и его миссии.
 
•	Высокоуровневые требования («система должна быть надёжной», «удобной для пользователей»).
 
•	Абстрактные бизнес цели («максимизировать конверсию», «минимизировать ошибки»).
 
•	Отсутствие конкретных сценариев, параметров, ограничений.
 
•	Общие принципы дизайна (например, «простота использования», «быстрая работа»).
 
•	Особенности: крайне мало конкретики, фокус на общих целях и принципах, требуется глубокая декомпозиция для генерации тест кейсов.
 
Пример:
 
Сервис помогает людям находить и бронировать лучшие отели быстро и просто.
Система должна быть надёжной и удобной.
Цель — сделать бронирование максимально простым и быстрым.
Пользователи ценят скорость и точность результатов.

Критерии качества входных данных
Для каждого набора должны выполняться следующие критерии:
 
•	Разнообразие типов: представлены разные виды документации (техническая, бизнес ориентированная, смешанная, абстрактная).
 
•	Контролируемая сложность: уровень «проблемности» (неполнота, противоречия, абстрактность) чётко задан для каждого набора.
