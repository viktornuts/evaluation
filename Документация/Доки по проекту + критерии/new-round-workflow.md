# Правило обработки команды "Новый раунд"

Документ фиксирует, что делает Codex/оценщик, когда пользователь присылает архив или набор файлов и пишет: **"Новый раунд"**.

## Триггер

Пользователь присылает:

- `zip`;
- `rar`;
- `7z`;
- набор отдельных файлов;
- папку с файлами;

и пишет: **"Новый раунд"**.

Первый ответ оценщика:

```text
Запускаю раунд №<номер>
```

Номер раунда определяется по текущей БД:

- смотрим существующие `eval_runs.run_code`;
- берем следующий номер после последнего раунда;
- если в файлах явно указан `run_code`, сверяем его с БД и предупреждаем о конфликте.

## Ожидаемый состав файлов

Для полного раунда ожидаются:

1. Файл версии агента.
2. Файл или файлы декомпозиции требований.
3. Файл или файлы сгенерированных тест-кейсов.

## Структура imports/exports по раундам

Каждый новый раунд должен иметь отдельные папки входа и выхода.

Входные файлы пользователя:

```text
imports/rounds/<run_code>/source/
```

Если пользователь прислал архив, оригинальный архив сохраняется в `source/`.
Распакованные файлы, если они нужны, складываются в:

```text
imports/rounds/<run_code>/extracted/
```

Выходные файлы по раунду:

```text
exports/rounds/<run_code>/
```

PDF-история по раунду:

```text
exports/pdf/history/<run_code>/eval_rounds_report_<run_code>.pdf
```

Актуальный сводный PDF:

```text
exports/pdf/eval_rounds_report_latest.pdf
```

Правило:

- новые файлы раунда не кладем напрямую в корень `imports/`;
- новые файлы раунда не кладем напрямую в корень `exports/`;
- файлы датасета не кладем напрямую в корень `imports/`, для них используем `imports/datasets/<dataset_code>_<version>/`;
- корень `imports/` остается для общих примеров, справочных импортов и README;
- корень `exports/` остается для кросс-раундовых артефактов.

Рекомендуемые имена:

```text
Версия агента.*
Декомпозиция_<dataset_code>.*
ТК_<dataset_code>.*
```

Для нескольких кейсов внутри датасета:

```text
Декомпозиция_<dataset_code>__<case_code>.*
ТК_<dataset_code>__<case_code>.*
```

Если файлы приходят в архиве, структура может быть такой:

```text
run_v3/
  Версия агента.xlsx
  customer_gold_v1/
    Декомпозиция_customer_gold_v1.xlsx
    ТК_customer_gold_v1.xlsx
  incomplete_requirements_v1/
    Декомпозиция_incomplete_requirements_v1.xlsx
    ТК_incomplete_requirements_v1.xlsx
```

## Действие 1. Проверка комплекта

Оценщик проверяет, что внутри прогона есть:

- информация о версии агента;
- декомпозиция для каждого датасета;
- тест-кейсы для каждого датасета;
- понятные `dataset_code` и `case_code`.

### Если нет файла версии агента

Оценщик спрашивает:

```text
Не вижу файл с версией агента. Идем дальше без него?
Если да, зафиксирую условные значения и отмечу это в отчете.
```

Если пользователь разрешает идти дальше:

- обязательные поля `eval_runs` заполняются условными значениями;
- в `change_summary` фиксируется, что данные об агенте неполные;
- в отчете явно пишется, что параметры агента не подтверждены.

### Если нет декомпозиции

Оценщик спрашивает:

```text
Не вижу файл с декомпозицией для <dataset_code>/<case_code>.
Идем дальше без декомпозиции? Тогда блок декомпозиции не попадет в оценку этого раунда.
```

Если пользователь разрешает идти дальше:

- данные по декомпозиции не импортируются;
- критерии декомпозиции остаются не измеренными;
- в отчете это фиксируется в выводе по блоку декомпозиции.

### Если нет тест-кейсов

Оценщик спрашивает:

```text
Не вижу файл с тест-кейсами для <dataset_code>/<case_code>.
Идем дальше без ТК? Тогда блок набора ТК и отдельных ТК не попадет в оценку этого раунда.
```

Если пользователь разрешает идти дальше:

- ТК не импортируются;
- оценки набора ТК и отдельных ТК не выставляются;
- в отчете это фиксируется.

## Действие 2. Разбор файлов и запись в БД

### 2.1. Файл "Версия агента"

Оценщик анализирует файл версии агента и раскладывает данные в таблицу:

```text
eval_runs
```

Обязательно ищем и фиксируем:

- версию агента;
- промпты;
- примеры-образцы, если они есть;
- модель;
- версию модели;
- `temperature`;
- `top_p`;
- режим работы: прямое извлечение / RAG / hybrid;
- параметры поиска, если есть;
- строгость формы ответа;
- краткое описание изменений;
- diff относительно прошлого раунда.

Ключевые поля БД:

```text
eval_runs.run_code
eval_runs.agent_name
eval_runs.agent_version
eval_runs.prompt_snapshot
eval_runs.model_name
eval_runs.model_version
eval_runs.temperature
eval_runs.top_p
eval_runs.run_mode
eval_runs.retrieval_chunk_size
eval_runs.retrieval_top_k
eval_runs.reranker_name
eval_runs.response_format_strictness
eval_runs.response_contract_json
eval_runs.change_summary
eval_runs.dataset_version
```

Правило по diff:

- если файл содержит явное описание изменений, переносим его в `change_summary`;
- если описания нет, оценщик формирует краткий diff сам на основе отличий промпта/настроек/модели от прошлого раунда;
- если прошлый раунд недоступен, пишем: `Первый известный раунд для данного набора данных`.

### 2.2. Файл "Декомпозиция_<dataset_code>"

Оценщик анализирует декомпозицию, сравнивает ее с эталонными атомарными требованиями и записывает результат в БД.

Основные таблицы:

```text
requirement_decomposition_evaluation_results
requirement_decomposition_quality_assessments
requirement_decomposition_quality_criteria
requirements
input_requirements
```

Критерии:

- `decomposition_completeness` - полнота декомпозиции;
- `decomposition_boundaries` - границы требований;
- `requirement_consolidation` - консолидация требований.

Для каждого критерия оценщик:

- выставляет `score`;
- выбирает/задает `label`;
- пишет `rationale`, почему поставлен именно такой балл;
- указывает `assessment_method`;
- при необходимости снижает `confidence`.

Если декомпозиция не передана, эти оценки не создаются.

### 2.3. Файл "ТК_<dataset_code>"

Оценщик анализирует тест-кейсы на двух уровнях:

1. Набор ТК целиком.
2. Каждый отдельный ТК.

Основные таблицы:

```text
test_cases
test_case_steps
requirement_test_case_links
test_case_step_requirement_links
test_suite_quality_assessments
test_case_quality_assessments
test_suite_quality_criteria
test_case_quality_criteria
```

Критерии набора ТК:

- `positive_coverage`;
- `negative_coverage`;
- `suite_cleanliness`;
- `required_checks_coverage`;
- `overall_completeness`.

Критерии отдельных ТК:

- `classification_correctness`;
- `template_required_attributes`;
- `conditions_quality`;
- `step_atomicity`;
- `expected_result_quality`;
- `no_hallucinations`.

Для каждого критерия оценщик:

- выставляет `score`;
- пишет `rationale`;
- связывает ТК с требованиями;
- отмечает спорные места как требующие ручного ревью.

## Действие 3. Формирование отчета

После записи данных в БД оценщик формирует:

1. PDF-отчет по раунду.
2. PDF-снимок истории нарастающим итогом.
3. PDF методологии оценок.

Команда:

```bash
python scripts/build_pdf_reports.py
```

### Правило сравнения в вердикте

Первые два раунда сравниваются с предыдущим раундом.

Начиная с третьего раунда, текущий раунд сравнивается с лучшей предыдущей версией - `TOP benchmark`.

`TOP benchmark` выбирается так:

1. максимальная общая полнота набора (`overall_completeness`);
2. если есть равенство - максимальная чистота набора (`suite_cleanliness`);
3. если есть равенство - максимальная достоверность отдельных ТК (`no_hallucinations`).

Вердикт должен показывать:

- что улучшилось относительно benchmark;
- что ухудшилось относительно benchmark;
- что осталось без существенных изменений;
- общий вывод, привело ли изменение агента к улучшению.

Правило динамики в вердикте:

- перед сравнением дельту округляем до одного знака;
- `>= +0.2` - улучшение;
- `<= -0.2` - ухудшение;
- от `-0.1` до `+0.1` - без существенных изменений.

Финальный общий вывод должен ссылаться только на реальные изменившиеся критерии. Если покрытие улучшилось, нельзя писать "просадки по покрытию"; нужно писать, по каким критериям действительно есть просадка, например: "при отдельных просадках по критериям: чистота набора, шаблон и обязательные атрибуты, атомарность шагов, достоверность". Если просадок нет, пишем: "без заметных просадок по измеряемым критериям".

Если текущая версия признается лучше benchmark, в выводе обязательно нужно объяснить причину выбора: какой ключевой критерий вырос и какие критерии стали драйверами улучшения. Формат: "версия лучше benchmark, потому что ключевой показатель ... вырос с ... до ...; рост обеспечен улучшением критериев ...; при этом есть ограничения ...". Не перечисляй просевшие критерии как будто они являются причиной выбора лучшей версии.

В отчете также должен быть отдельный блок `TOP benchmark`. Сначала нужно простым языком объяснить, что TOP benchmark - это лучшая предыдущая версия агента по текущей методике оценки. Затем нужно показать:

- текущий раунд;
- с каким benchmark он сравнивается;
- почему выбран именно этот benchmark: общая полнота набора, чистота набора, достоверность отдельных ТК;
- итоговую фразу вида: "Текущий раунд `<run_code>` сравнивается с `<benchmark_run_code>`, которая сейчас является TOP benchmark".

Актуальный отчет:

```text
exports/pdf/eval_rounds_report_latest.pdf
```

История конкретного раунда:

```text
exports/pdf/history/<run_code>/eval_rounds_report_<run_code>.pdf
```

Markdown/служебные отчеты конкретного раунда:

```text
exports/rounds/<run_code>/
```

Методология:

```text
exports/pdf/methodology/scoring_methodology.pdf
```

## Что отправляем пользователю в чат

После завершения раунда оценщик пишет:

- какой номер раунда обработан;
- какие файлы были найдены;
- каких файлов не хватало;
- какие данные попали в БД;
- какие блоки отчета измерены;
- какие блоки не измерены;
- ссылки на PDF-отчет и PDF-методологию.

Минимальный финальный формат:

```text
Раунд №<номер> обработан.

В БД записано:
- версия агента: да/нет;
- декомпозиция: да/нет;
- ТК: да/нет;
- оценки декомпозиции: да/нет;
- оценки набора ТК: да/нет;
- оценки отдельных ТК: да/нет.

Отчет:
<ссылка на PDF>

Методология:
<ссылка на PDF>
```

## Важное правило

Если каких-то данных нет, оценщик не должен молча придумывать их как подтвержденные.

Допустимо:

- продолжить с условными значениями после подтверждения пользователя;
- явно отметить неполноту в `change_summary`;
- явно написать в отчете, какие блоки не измерялись.

Недопустимо:

- выдавать непереданную декомпозицию за измеренную;
- выдавать отсутствующие параметры агента за реальные;
- смешивать разные датасеты без `dataset_code` и `case_code`;
- перетирать историю старых раундов без явной причины.
