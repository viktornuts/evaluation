# Методика выставления оценок по критериям

Документ описывает, как интерпретировать оценки по шкале `0..10` для MVP eval-процесса.
Оценка хранится как `REAL`, поэтому допустимы дробные значения: `6.5`, `7.2`, `8.5` и т.д.

## Общий принцип шкалы

| Оценка | Интерпретация |
|---:|---|
| `10` | Эталон: критерий выполнен полностью, замечаний нет. |
| `8-9` | Почти эталон: есть небольшие замечания, но они не ломают результат. |
| `6-7` | Частично хорошо: основная польза есть, но есть заметные пропуски или спорные места. |
| `4-5` | Слабо: критерий выполнен фрагментарно, качество заметно мешает использовать результат. |
| `1-3` | Плохо: результат почти не соответствует критерию. |
| `0` | Критерий не выполнен или сущность отсутствует. |

Дробные оценки используются, когда результат находится между двумя целыми значениями.
Например, `6.5` означает: результат лучше, чем типичная `6`, но еще не дотягивает до уверенной `7`.

Пример:

- `6.0` - основная часть покрыта, но пропуски существенные.
- `6.5` - основная часть покрыта, часть важных пропусков остается.
- `7.0` - покрытие все еще неполное, но пропуски уже не критичные для демонстрационного результата.

## 1. Сгенерированные требования / декомпозиция

Эти критерии применяются к требованиям, которые агент получил после декомпозиции исходного материала.
Сравнение идет с эталонными атомарными требованиями.

### `decomposition_completeness` - Полнота декомпозиции

Проверяет, все ли значимые атомарные требования из эталона были выделены агентом.

Как считать:

```text
score = (количество найденных значимых атомарных требований / количество эталонных атомарных требований) * 10
```

После расчета оценка может быть скорректирована вниз, если требование формально найдено, но искажено по смыслу.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Все значимые атомарные требования выделены, потерь нет. |
| `8-9` | Почти все требования выделены, есть небольшие пропуски. |
| `6-7` | Выделена основная часть требований, но есть заметные пропуски. |
| `4-5` | Потеряна существенная часть требований или важных ограничений. |
| `1-3` | Выделены единичные требования, большая часть смысла потеряна. |
| `0` | Декомпозиция отсутствует или не отражает входное требование. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если агент выделил основу, но пропустил несколько важных атомарных требований, из-за чего результат еще нельзя считать уверенно близким к приемлемому.

### `decomposition_boundaries` - Границы требований

Проверяет, корректно ли агент нарезал требования: не склеил независимые требования и не раздробил одно требование на бессмысленные части.

Как считать:

```text
score = экспертная оценка корректности границ по шкале 0..10
```

При наличии точной разметки можно считать долю требований с корректными границами:

```text
score = (количество требований с корректными границами / количество проверенных требований) * 10
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Все атомарные требования выделены самостоятельными проверяемыми единицами. |
| `8-9` | Границы в целом корректны, есть небольшие спорные места. |
| `6-7` | Часть требований нарезана корректно, но есть заметные склейки или лишнее дробление. |
| `4-5` | Границы часто нарушены, атомарность заметно потеряна. |
| `1-3` | Большая часть декомпозиции состоит из некорректных фрагментов. |
| `0` | Границы не определены или декомпозиция непригодна. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если большая часть требований нарезана понятно, но есть несколько важных случаев склейки/разрыва, которые повлияют на генерацию ТК.

### `requirement_consolidation` - Консолидация требований

Проверяет, корректно ли агент объединяет одинаковую или связанную информацию из разных источников: без дублей, противоречий и потери смысла.

Как считать:

```text
score = экспертная оценка консолидации по шкале 0..10
```

При наличии точной разметки можно считать:

```text
score = (количество корректно объединенных связок / количество связок, которые нужно было объединить) * 10
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Связанная информация объединена корректно, дублей и противоречий нет. |
| `8-9` | Консолидация в целом корректна, есть небольшие дубли или потери контекста. |
| `6-7` | Часть информации объединена корректно, часть продублирована или требует ревью. |
| `4-5` | Есть заметные дубли, несогласованные формулировки или потеря связей. |
| `1-3` | Агент почти не консолидирует связанные требования. |
| `0` | Консолидация отсутствует или вводит в заблуждение. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если агент частично справился с объединением источников, но оставил дубли или потерял часть связей, которые важны для последующих ТК.

## 2. Набор тест-кейсов

Эти критерии применяются ко всему набору ТК, который агент сгенерировал по требованиям.
Оценка показывает качество набора как целого, а не качество одного конкретного ТК.

### `positive_coverage` - Позитивное покрытие

Проверяет, есть ли позитивные тест-кейсы для требований, где позитивный сценарий нужен.

Как считать:

```text
score = (количество требований с релевантным позитивным ТК / количество требований, где позитивный ТК нужен) * 10
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Все применимые требования покрыты позитивными ТК. |
| `8-9` | Покрытие почти полное, есть небольшие пропуски. |
| `6-7` | Позитивные сценарии есть для основной части требований. |
| `4-5` | Позитивные сценарии есть выборочно и не закрывают скоуп. |
| `1-3` | Позитивное покрытие почти отсутствует. |
| `0` | Позитивные ТК отсутствуют там, где они нужны. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если позитивные проверки закрывают основную часть требований, но есть важные непокрытые требования или покрытие слишком общее.

### `negative_coverage` - Негативное покрытие

Проверяет, есть ли негативные тест-кейсы там, где требование подразумевает ошибки, запреты, ограничения или альтернативное поведение.

Как считать:

```text
score = (количество требований с релевантным негативным ТК / количество требований, где негативный ТК нужен) * 10
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Все применимые требования покрыты негативными ТК. |
| `8-9` | Негативное покрытие почти полное, есть небольшие пропуски. |
| `6-7` | Негативные сценарии есть для основной части применимых требований. |
| `4-5` | Негативные сценарии есть выборочно и не закрывают основные риски. |
| `1-3` | Негативное покрытие почти отсутствует. |
| `0` | Негативные ТК отсутствуют там, где они нужны. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если негативные проверки есть, но часть из них производная, спорная или не подтверждена требованиями.

### `suite_cleanliness` - Чистота набора

Проверяет, нет ли в наборе лишних, нерелевантных, дублирующих или галлюцинированных ТК.

Как считать:

Для MVP используется экспертная оценка по шкале `0..10` с учетом количества и серьезности проблем.
Можно применять ориентировочную формулу:

```text
score = 10 - штрафы за дубли, лишние ТК, нерелевантные ТК и проверки вне источников
```

Пример штрафов:

- небольшой дубль или спорный ТК: `-0.5..-1`;
- явный дубль: `-1..-2`;
- нерелевантный ТК: `-2..-3`;
- ТК на неподтвержденную функциональность: `-2..-4`.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | В наборе нет лишних, нерелевантных и дублирующих ТК. |
| `8-9` | Есть единичные незначимые дубли или спорные ТК. |
| `6-7` | Есть заметные дубли или спорные ТК, но набор в целом полезен. |
| `4-5` | Лишние или дублирующие ТК существенно загрязняют набор. |
| `1-3` | Набор в основном состоит из дублей или нерелевантных ТК. |
| `0` | Набор непригоден из-за лишних, дублирующих или нерелевантных ТК. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если набор полезен, но в нем уже достаточно дублей/лишних сценариев, чтобы это мешало анализу.

### `required_checks_coverage` - Покрытие обязательных проверок

Проверяет, покрыты ли обязательные проверки из требований или эталонного набора.

Как считать:

```text
score = (количество покрытых обязательных проверок / количество обязательных проверок всего) * 10
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Все обязательные проверки покрыты. |
| `8-9` | Почти все обязательные проверки покрыты. |
| `6-7` | Покрыта основная часть обязательных проверок. |
| `4-5` | Покрыты отдельные обязательные проверки, но есть существенные пропуски. |
| `1-3` | Большинство обязательных проверок пропущено. |
| `0` | Обязательные проверки не покрыты. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если формально покрыта основная часть обязательных проверок, но среди пропусков есть важные проверки. Например, покрыто около 65% обязательных проверок или покрытие ближе к 70%, но часть проверок спорная.

### `overall_completeness` - Общая полнота набора

Проверяет, достаточен ли набор ТК для проверяемого скоупа в целом.

Как считать:

Это итоговая экспертная оценка по набору. Она учитывает:

- позитивное покрытие;
- негативное покрытие;
- обязательные проверки;
- чистоту набора;
- наличие основных бизнес/API/UI/интеграционных сценариев;
- пригодность набора для ручной или автоматизированной проверки.

В MVP это не простое среднее арифметическое, потому что один критичный пропуск может быть важнее нескольких мелких замечаний.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Набор достаточен для скоупа и закрывает основные ожидаемые сценарии. |
| `8-9` | Набор почти достаточен, есть небольшие пропуски. |
| `6-7` | Набор покрывает основу, но требует доработки. |
| `4-5` | Набор недостаточен для уверенной проверки скоупа. |
| `1-3` | Набор почти не закрывает проверяемый скоуп. |
| `0` | Набор отсутствует или непригоден. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если набор уже демонстрирует рабочее покрытие, но требует заметной ручной доработки перед использованием как приемочного результата.

## 3. Отдельный тест-кейс

Эти критерии применяются к каждому отдельному ТК. В итоговом отчете по раунду может выводиться средняя оценка по всем ТК.

### `classification_correctness` - Корректность вида ТК

Проверяет, правильно ли определен вид ТК: UI, API, интеграционный или другой применимый тип.

Как считать:

```text
score = экспертная оценка соответствия типа содержанию ТК
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Вид ТК полностью соответствует сценарию. |
| `8-9` | Вид в целом корректен, есть небольшая спорность. |
| `6-7` | Вид вероятно корректен, но требует проверки. |
| `4-5` | Вид слабо соответствует сценарию. |
| `1-3` | Вид скорее неверный. |
| `0` | Вид отсутствует или полностью неверен. |

### `template_required_attributes` - Шаблон и обязательные атрибуты

Проверяет, соответствует ли ТК шаблону и заполнены ли обязательные атрибуты: ссылка на задачу, ссылка на требования, тип, направление, заголовок и другие обязательные поля.

Как считать:

```text
score = (количество заполненных обязательных атрибутов / количество обязательных атрибутов) * 10
```

Если структура ТК нарушена настолько, что атрибуты нельзя надежно выделить, оценка дополнительно снижается.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Шаблон соблюден, все обязательные атрибуты заполнены. |
| `8-9` | Шаблон почти соблюден, есть небольшие пробелы. |
| `6-7` | Основная структура есть, часть обязательных атрибутов заполнена неполно. |
| `4-5` | Шаблон или обязательные атрибуты существенно нарушены. |
| `1-3` | ТК почти не соответствует шаблону. |
| `0` | Шаблон и обязательные атрибуты отсутствуют. |

### `conditions_quality` - Предусловия и постусловия

Проверяет, есть ли у ТК корректные предусловия и постусловия, и не записаны ли постусловия как шаги сценария.

Как считать:

```text
score = экспертная оценка полноты и корректности предусловий/постусловий
```

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Предусловия и постусловия корректно описаны и соответствуют сценарию. |
| `8-9` | В целом корректны, есть небольшие пробелы. |
| `6-7` | Частично описаны, часть контекста требует уточнения. |
| `4-5` | Неполные, спорные или плохо связаны со сценарием. |
| `1-3` | Почти потеряны или сформулированы как шаги. |
| `0` | Необходимые предусловия и постусловия отсутствуют. |

### `step_atomicity` - Атомарность шагов

Проверяет, что каждый шаг содержит одно действие или одну проверяемую операцию.

Как считать:

```text
score = (количество атомарных шагов / количество шагов всего) * 10
```

Если шаги формально атомарны, но сценарий из-за этого потерял смысл, оценка корректируется экспертно.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Каждый шаг содержит одно действие или одну проверяемую операцию. |
| `8-9` | Шаги почти атомарны, есть единичные небольшие склейки. |
| `6-7` | Часть шагов смешивает несколько действий, но сценарий остается понятным. |
| `4-5` | Много шагов объединяют разные действия или проверки. |
| `1-3` | Шаги в основном неатомарны и требуют переразбиения. |
| `0` | Шаги отсутствуют или не являются шагами тестового сценария. |

### `expected_result_quality` - Ожидаемый результат

Проверяет, что у каждого шага есть конкретный, проверяемый и соответствующий шагу ожидаемый результат.

Как считать:

```text
score = (количество шагов с корректным ожидаемым результатом / количество шагов всего) * 10
```

Ожидаемый результат считается некорректным, если он слишком общий, не проверяемый или не соответствует действию в шаге.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | У каждого шага есть конкретный и проверяемый ожидаемый результат. |
| `8-9` | Ожидаемые результаты почти все корректны, есть небольшие неточности. |
| `6-7` | Часть ожидаемых результатов неполная, но основной сценарий проверяем. |
| `4-5` | Ожидаемые результаты часто общие, непроверяемые или плохо связаны с шагами. |
| `1-3` | Ожидаемые результаты почти отсутствуют или не соответствуют действиям. |
| `0` | Ожидаемые результаты отсутствуют. |

### `no_hallucinations` - Достоверность / отсутствие галлюцинаций

Проверяет, что ТК не содержит фактов, деталей, UI-элементов, ролей, API, статусов, endpoint-ов или ограничений, которых нет в требованиях и источниках.

Как считать:

Для MVP используется экспертная оценка с учетом количества и серьезности неподтвержденных фактов.

Ориентировочно:

```text
score = 10 - штрафы за неподтвержденные факты
```

Пример штрафов:

- мелкая неподтвержденная деталь без влияния на сценарий: `-0.5..-1`;
- спорная роль, поле, статус или UI-элемент: `-1..-2`;
- неподтвержденный endpoint, бизнес-правило или ограничение: `-2..-4`;
- ТК построен на выдуманном поведении: оценка уходит в диапазон `1-3` или `0`.

Ориентиры:

| Оценка | Когда ставим |
|---:|---|
| `10` | Неподтвержденных фактов нет. |
| `8-9` | Есть небольшие детали с низким риском неподтвержденности. |
| `6-7` | Есть отдельные неподтвержденные факты, требуется ревью. |
| `4-5` | Неподтвержденные факты заметно влияют на корректность ТК. |
| `1-3` | ТК в основном построен на выдуманных фактах. |
| `0` | ТК непригоден из-за галлюцинаций. |

Почему может быть `6.5`, а не `7`:

`6.5` ставим, если ТК в целом проверяет нужное требование, но содержит несколько неподтвержденных деталей, которые нужно отдельно проверить руками.

## Как читать итоговую оценку в отчете

Для набора ТК оценка обычно выставляется один раз на весь набор.

Для отдельных ТК оценка выставляется каждому тест-кейсу отдельно, а в итоговом отчете по раунду выводится среднее значение по всем ТК:

```text
score_in_report = среднее значение оценок по всем ТК по выбранному критерию
```

Например, если по критерию `expected_result_quality` часть ТК получила `10`, часть `8`, а часть `6`, итоговая оценка в отчете может быть `8.5/10`.

## Почему оценка может быть дробной

Дробная оценка нужна в двух случаях:

1. Оценка рассчитана по доле покрытия.

Пример:

```text
13 покрытых обязательных проверок / 20 обязательных проверок * 10 = 6.5
```

2. Экспертная оценка находится между двумя уровнями шкалы.

Пример:

```text
6 = основная часть покрыта, но пропуски существенные
7 = основная часть покрыта, пропуски уже умеренные
6.5 = ситуация между этими состояниями
```

Таким образом, `6.5` - это не случайная оценка, а способ показать промежуточное качество между двумя описанными уровнями.
