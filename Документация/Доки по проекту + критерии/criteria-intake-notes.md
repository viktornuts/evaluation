## Что в файлах

### Критерии от Екатерины / Павла

Файлы содержат одинаковый набор критериев для проверки работы модели:

- корректно определить количество требований из документа-источника;
- корректно определить границы требований;
- объединять информацию по одному требованию из разных источников;
- не дублировать требования и сгенерированные по ним ТК;
- различать user cases и технические требования;
- корректно классифицировать требования и ТК по видам;
- не генерировать дополнительные виды ТК, не соответствующие требованию;
- использовать шаблон ТК, соответствующий виду ТК;
- копировать одинаковые prerequisites в каждое требование, где они нужны;
- генерировать минимум один позитивный и один негативный ТК на требование;
- не терять предусловия;
- заполнять обязательные атрибуты ТК;
- генерировать релевантные ТК;
- корректно разбивать сценарий на шаги;
- обеспечивать соответствие ожидаемого результата шагу;
- корректно формировать постусловия;
- не галлюцинировать и не выдумывать факты, отсутствующие в требованиях.

### Критерии требований.txt

Содержит 9 критериев анализа требований:

- `correctness` - корректность;
- `unambiguity` - однозначность;
- `completeness` - полнота;
- `consistency` - непротиворечивость;
- `testability` - тестируемость;
- `traceability` - прослеживаемость;
- `modifiability` - модифицируемость;
- `atomicity` - атомарность;
- `feasibility` - выполнимость.

В текущей БД уже есть близкие критерии требований:

- `source_quality`
- `completeness`
- `consistency`
- `correctness`
- `ambiguity`
- `testability`
- `traceability`

Нужно отдельно решить, добавляем ли:

- `modifiability`
- `atomicity`
- `feasibility`

Также нужно решить, оставляем ли текущий код `ambiguity` или переименовываем/дублируем как `unambiguity`. Сейчас логика такая: высокий score по `ambiguity` означает хорошую недвусмысленность, но название может путать.

### Критерии Проверок.txt

Описывает не конкретные требования/ТК, а типы входных наборов для проверки устойчивости модели:

- набор 1: структурированная и полная техническая документация;
- набор 2: фрагментированная и неполная документация;
- набор 3: бизнес-ориентированная документация с user stories;
- набор 4: смешанная документация с противоречиями и шумом;
- набор 5: высокоуровневая и абстрактная документация.

Также задает общие ограничения для входных наборов:

- только текст;
- объем 300-800 слов;
- русский язык с допустимыми техническими терминами на английском;
- нельзя добавлять явные подсказки для генерации тест-кейсов;
- вход должен выглядеть как реальная документация, а не как задание ассистенту.

Это похоже на отдельный слой критериев качества `dataset_case` / входного набора. В текущей БД отдельной таблицы под такие критерии пока нет.

## Предварительное сопоставление с БД

### Уже хорошо ложится

- K4-K8 -> `test_case_quality_criteria` / `test_case_quality_assessments`
- K2-K3 -> `test_suite_quality_criteria` / `test_suite_quality_assessments`
- K9 -> частично `test_case_quality_criteria`, `test_suite_quality_criteria`, `unsupported_details`, `requirement_test_case_links`
- критерии требований -> `requirement_quality_criteria` / `requirement_quality_assessments`

### Нужна доработка модели

1. K1 про полноту извлечения, границы и консолидацию требований - это не просто качество отдельного требования. Это результат сравнения extracted requirements с эталонным списком. Для этого позже нужна таблица уровня `requirement_evaluation_results` или `requirement_extraction_evaluation_results`.

2. Критерии входных наборов из `Критерии Проверок.txt` не являются критериями требований или ТК. Решение от 2026-06-22: не заводим отдельные таблицы, а храним профиль входного набора прямо в `dataset_cases.input_profile_code` / `dataset_cases.input_profile_name`.

3. В критериях требований были новые пункты:

```text
modifiability
atomicity
feasibility
```

Их нужно добавить в `requirement_quality_criteria`, если принимаем этот список как основной.

Решение от 2026-06-22: принимаем список требований как основной и добавляем `modifiability`, `atomicity`, `feasibility`. Также заменяем код `ambiguity` на `unambiguity`, чтобы высокий score означал хорошую однозначность и название не путало.

## Рекомендуемый порядок наполнения

1. Сначала финализировать `requirement_quality_criteria` для требований.
2. Потом финализировать `test_case_quality_criteria` для отдельного ТК.
3. Потом финализировать `test_suite_quality_criteria` для набора ТК.
4. После этого решить, нужна ли отдельная таблица критериев входного набора (`dataset_case_quality_criteria`).
5. Затем взять `Эталонные требования.png` или исходную таблицу, если есть Excel/CSV, и занести первый реальный набор требований.
