# Правила формирования отчетов

## Формат выдачи

Основной формат выдачи для демонстрации и согласования - PDF.

Актуальные файлы:

```text
exports/pdf/eval_rounds_report_latest.pdf
exports/pdf/methodology/scoring_methodology.pdf
```

## История по раундам

Для каждого нового eval-раунда формируется отдельный PDF-снимок с нарастающим итогом:

```text
exports/pdf/history/<run_code>/eval_rounds_report_<run_code>.pdf
```

Пример:

```text
exports/pdf/history/v0/eval_rounds_report_v0.pdf
exports/pdf/history/v1/eval_rounds_report_v1.pdf
exports/pdf/history/v2/eval_rounds_report_v2.pdf
```

Исторические PDF не перетираются вручную. Они нужны, чтобы можно было открыть состояние отчета на момент конкретного раунда.

## Методология оценок

Каждый PDF-отчет должен ссылаться на методологию:

```text
exports/pdf/methodology/scoring_methodology.pdf
```

Методология хранится в Markdown как исходник:

```text
Документация/Доки по проекту + критерии/scoring-methodology.md
```

PDF формируется командой:

```bash
python scripts/build_pdf_reports.py
```

## Пояснения к оценкам

В отчете должны быть пояснения под каждым основным блоком:

- декомпозиция требований;
- набор ТК;
- отдельные ТК.

Пояснение должно отвечать на вопрос: почему по каждому критерию выставлен именно такой балл.

## Benchmark для сравнения

Первые два раунда сравниваются последовательно:

```text
v1 сравнивается с v0
```

Начиная с третьего раунда, текущий раунд сравнивается не обязательно с предыдущим, а с лучшей предыдущей версией - `TOP benchmark`.

Правило выбора `TOP benchmark`:

1. Берем все предыдущие раунды по этому датасету.
2. Выбираем раунд с максимальной общей полнотой набора (`overall_completeness`).
3. Если есть равенство, выбираем тот, у кого выше чистота набора (`suite_cleanliness`).
4. Если снова равенство, выбираем тот, у кого выше достоверность отдельных ТК (`no_hallucinations`).

Так мы не теряем ситуацию, когда новая версия лучше предыдущей, но все еще хуже более ранней TOP-версии.

В вердикте отчета нужно писать:

- с какой версией сравниваем текущий раунд;
- что улучшилось относительно benchmark;
- что ухудшилось относительно benchmark;
- общий вывод, к чему привело изменение агента.

Правило классификации изменений:

- дельту считаем после округления до одного знака;
- изменение `>= +0.2` считается улучшением;
- изменение `<= -0.2` считается ухудшением;
- изменение от `-0.1` до `+0.1` считается без существенной динамики.

В общем выводе нельзя писать обобщенную причину, если она не совпадает с фактическими изменившимися критериями. Например, нельзя писать "просадки по покрытию", если покрытие улучшилось, а ухудшились чистота, шаблон или достоверность. Нужно называть реальные ухудшившиеся критерии: "при отдельных просадках по критериям: чистота набора, шаблон и обязательные атрибуты, достоверность". Если ухудшений нет, нужно писать "без заметных просадок по измеряемым критериям".

Если в общем выводе написано, что текущая версия лучше benchmark, обязательно нужно объяснить, за счет чего она признана лучше. Сначала указываем ключевой критерий выбора/улучшения, например: "общая полнота набора выросла с 8.5 до 8.7". Затем перечисляем критерии-драйверы улучшения: позитивное покрытие, покрытие обязательных проверок и т.п. Только после этого перечисляем ограничения и просевшие критерии. Просевшие критерии нельзя подавать как причину выбора лучшей версии.

В PDF-отчете должен быть отдельный блок `TOP benchmark` до основных таблиц. Сначала нужно простыми словами объяснить, что TOP benchmark - это лучшая предыдущая версия агента по текущей методике оценки. После этого нужно явно показать:

- текущий раунд;
- выбранный benchmark run;
- режим сравнения;
- значения, по которым benchmark выбран, человекочитаемыми названиями: общая полнота набора, чистота набора, достоверность отдельных ТК;
- итоговую фразу вида: "Текущий раунд `<run_code>` сравнивается с `<benchmark_run_code>`, которая сейчас является TOP benchmark".
