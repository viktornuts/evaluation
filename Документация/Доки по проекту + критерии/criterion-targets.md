# Целевые показатели критериев

Файл фиксирует целевые значения для отчетов по прогонам. Эти же цели хранятся в БД в полях:

- `target_score`
- `target_display`
- `target_description`

Для разных типов датасетов цель может отличаться. Базовые цели ниже используются как fallback.
Профильные и частные цели хранятся в БД:

- `dataset_input_profiles` - справочник профилей входных данных;
- `dataset_profile_criterion_targets` - цели по профилю датасета;
- `dataset_case_criterion_targets` - переопределения для конкретного dataset case.

Порядок выбора цели:

1. цель конкретного `dataset_case_id`;
2. цель профиля `input_profile_code`;
3. базовая цель из таблицы критерия.

## Входные требования

| Код в БД | Цель |
|---|---|
| `source_quality` | 10 / подтвержденный источник |
| `completeness` | 10 / полное требование |
| `consistency` | 10 / без противоречий |
| `correctness` | 10 / корректно |
| `unambiguity` | 10 / однозначно |
| `testability` | 10 / проверяемо |
| `traceability` | 10 / трассируемо |
| `modifiability` | 10 / поддерживаемо |
| `atomicity` | 10 / атомарно |
| `feasibility` | 10 / выполнимо |

## Профили датасетов

| Профиль | Что означает 10/10 |
|---|---|
| `customer_gold_requirements` | Максимально близкое совпадение с эталонными требованиями и эталонными ТК заказчика. |
| `complete_technical_docs` | Полное покрытие требований и обязательных проверок без лишних сценариев. |
| `incomplete_fragmented_docs` | Безопасное частичное покрытие: покрыто только то, что выводимо из источников; пробелы отмечены; галлюцинаций нет. |
| `business_user_story_docs` | Покрыто проверяемое бизнес-ядро без выдуманной технической детализации. |
| `conflicting_noisy_docs` | Шум отфильтрован, противоречия не склеены в ложное уверенное требование, лишние ТК не сгенерированы. |
| `abstract_high_level_docs` | Покрыто проверяемое ядро, недостающие детали отмечены как пробелы, неподтвержденные частности не добавлены. |

## Как читать цель для плохих требований

Для датасета с плохими или неполными требованиями `10/10` не означает, что агент обязан получить такой же полный набор ТК, как по эталонной документации.

`10/10` означает, что агент сделал лучший корректный результат для данного качества входа:

- не потерял явно заданные требования;
- не придумал отсутствующие детали;
- отделил проверяемое от непроверяемого;
- корректно отметил пробелы, конфликты или неоднозначность;
- сформировал ТК только по подтвержденным и выводимым сценариям.

Если нужно задать особую цель для конкретного датасета, используем `dataset_case_criterion_targets`. Например, можно указать, что для `BADREQ-001` ожидается не 100% полное покрытие, а 100% покрытие проверяемого ядра с явной фиксацией непокрытых из-за нехватки данных мест.
