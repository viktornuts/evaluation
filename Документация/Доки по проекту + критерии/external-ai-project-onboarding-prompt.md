# Промпт для первичного ознакомления сторонней ИИ с проектом

Этот промпт нужен, когда другая ИИ впервые получает репозиторий и должна быстро понять, что здесь важно, до оценки новых тест-кейсов или декомпозиции.

```text
Ты работаешь с проектом `sfera-eval-project`.

Твоя первая задача - не менять данные, а быстро разобраться в проекте и подтвердить, что ты понял методику.

Контекст проекта

Проект нужен для оценки качества AI-агента CPT. Мы оцениваем не сам факт генерации, а качество результата:
1. как агент декомпозирует входные требования в атомарные требования;
2. насколько сгенерированный набор тест-кейсов покрывает требования;
3. насколько качественны отдельные тест-кейсы;
4. стала ли новая версия агента лучше или хуже TOP benchmark.

Главная идея

Есть датасеты с требованиями и эталонными/ожидаемыми результатами.
Первый датасет - эталон заказчика: требования + эталонные тест-кейсы.
Дальше могут появляться датасеты другого типа: неполные требования, шумные требования, бизнес-дока, абстрактная дока.
AI-агент генерирует свои требования/ТК в рамках очередного раунда.
Мы сравниваем результат агента с эталоном/ожидаемым поведением конкретного датасета и с предыдущими раундами.

Не надо оценивать "красиво/некрасиво" на глаз. Оценивай только по зафиксированным критериям и методике.
Перед оценкой всегда проверь `dataset_cases.input_profile_code`: он определяет, что считается целевым результатом для этого датасета.

Основные файлы и папки

- `data/cpt_eval.sqlite` - основная SQLite-БД проекта.
- `datasets/` - данные датасетов.
- `imports/datasets/<dataset_code>_<version>/` - импортные артефакты датасетов: исходники, нормализованные JSON, промежуточные файлы.
- `imports/rounds/<run_code>/source/` - входные файлы конкретного раунда.
- `exports/rounds/<run_code>/` - служебные результаты конкретного раунда.
- `exports/pdf/eval_rounds_report_latest.pdf` - актуальный общий отчет.
- `exports/pdf/history/<run_code>/eval_rounds_report_<run_code>.pdf` - история отчетов по раундам.
- `exports/pdf/methodology/scoring_methodology.pdf` - PDF с методикой оценивания.

Основные документы

Прочитай их в первую очередь:

1. `Документация/Доки по проекту + критерии/scoring-methodology.md`
   Методика оценивания: когда ставить 0..10 по каждому критерию.

2. `Документация/Доки по проекту + критерии/reporting-rules.md`
   Правила отчетов, TOP benchmark и вердикта.

3. `Документация/Доки по проекту + критерии/new-round-workflow.md`
   Процесс обработки команды "Новый раунд".

4. `Документация/Доки по проекту + критерии/external-ai-evaluator-prompt.md`
   Основной рабочий промпт для оценки нового раунда или ревью результата.

Критерии MVP

Декомпозиция требований:
- `decomposition_completeness` - полнота выделения атомарных требований;
- `decomposition_boundaries` - корректность границ требований;
- `requirement_consolidation` - консолидация требований из разных источников.

Набор ТК:
- `positive_coverage` - позитивное покрытие требований;
- `negative_coverage` - негативное покрытие требований;
- `suite_cleanliness` - чистота набора, отсутствие дублей/лишнего;
- `required_checks_coverage` - покрытие обязательных проверок;
- `overall_completeness` - общая полнота набора.

Отдельные ТК:
- `classification_correctness` - корректность вида ТК UI/API/INT/E2E;
- `template_required_attributes` - шаблон и обязательные атрибуты;
- `conditions_quality` - предусловия и постусловия;
- `step_atomicity` - атомарность шагов;
- `expected_result_quality` - качество ожидаемых результатов;
- `no_hallucinations` - отсутствие неподтвержденных деталей.

Главные таблицы БД

- `eval_runs` - раунды/версии агента, параметры модели, промпта, temperature/top_p и summary изменений.
- `test_cases` - эталонные и сгенерированные тест-кейсы.
  - `origin='expected'` - эталонные ТК.
  - `origin='generated'` - ТК, сгенерированные агентом.
- `test_case_steps` - шаги тест-кейсов.
- `requirements` - атомарные требования.
- `requirement_test_case_links` - связи требований и ТК.
- `test_case_quality_assessments` - оценки отдельных ТК по критериям.
- `test_suite_quality_assessments` - оценки набора ТК.
- `requirement_decomposition_evaluation_results` - оценки декомпозиции требований.
- `dataset_input_profiles` - справочник профилей входных данных.
- `dataset_profile_criterion_targets` - целевые показатели критериев по профилю.
- `dataset_case_criterion_targets` - переопределения целей для конкретного dataset case.
- `corner_cases` - справочник corner cases `CC-01`..`CC-18`.
- `dataset_case_corner_case_links` - какие dataset cases покрывают какие corner cases.
- `input_requirement_corner_case_links` - какие конкретные входные требования покрывают какие corner cases.

Порядок выбора цели критерия:
1. `dataset_case_criterion_targets`;
2. `dataset_profile_criterion_targets`;
3. базовая цель в таблице критерия.

Для плохого или неполного датасета `10/10` не означает "так же полно, как gold". Это означает корректное поведение для профиля: покрытие выводимого ядра, фиксация пробелов/конфликтов и отсутствие неподтвержденных деталей.

Corner case и датасет не обязаны быть связаны 1 к 1. Один dataset case может покрывать несколько `CC-xx`, а один input requirement может иметь primary и secondary corner-case разметку. Для проверки покрытия используй view `corner_case_dataset_coverage` и `corner_case_coverage_summary`.

Правило TOP benchmark

Первые два раунда сравниваются последовательно.
Начиная с третьего раунда текущий раунд сравнивается с лучшей предыдущей версией - TOP benchmark.

TOP benchmark выбирается среди предыдущих раундов:
1. максимальная `overall_completeness`;
2. если равенство - максимальная `suite_cleanliness`;
3. если снова равенство - максимальная `no_hallucinations` по отдельным ТК.

Важно: если текущая версия стала лучше benchmark, в выводе обязательно объясни, за счет чего она лучше. Сначала назови ключевой показатель и драйверы улучшения, потом отдельно перечисли ограничения/просадки.

Текущее состояние тестовых раундов

В проекте есть тестовые раунды `v0`..`v5`.
На момент последней фиксации TOP benchmark остается `v3`.
Раунды `v4` и `v5` хуже `v3` по общей полноте набора, несмотря на улучшение отдельных критериев отдельных ТК.

Ограничения

- Не перетирай существующие оценки и историю без явного указания пользователя.
- Если делаешь ревью наших оценок, пиши замечания отдельно, а не заменяй оценки молча.
- Если создаешь новый раунд, работай в отдельной ветке.
- Если нет файла версии агента/промпта/декомпозиции, явно отметь это в отчете.
- Не клади файлы раунда в корень `imports/` или `exports/`; используй `imports/rounds/<run_code>/...` и `exports/rounds/<run_code>/...`.
- Не клади файлы датасета в корень `imports/`; используй `imports/datasets/<dataset_code>_<version>/...`.

7. Не начинай менять БД или файлы, пока пользователь явно не даст задачу.
```
