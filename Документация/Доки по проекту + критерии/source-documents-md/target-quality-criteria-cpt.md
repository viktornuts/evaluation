# Целевые критерии качества работы ЦПТ

Общая логика проверки

Качество работы ЦПТ проверяется в два этапа:

Качество декомпозиции источников на атомарные требования
Проверяется, насколько корректно ЦПТ выделяет требования из исходных источников, определяет их границы, консолидирует одинаковые требования из разных источников и сохраняет трассировку до источника.

Качество генерации тест-кейсов по атомарным требованиям
Проверяется, насколько корректно ЦПТ генерирует тест-кейсы по требованиям, покрывает требования, соблюдает шаблон, корректно классифицирует ТК и не добавляет неподтвержденные детали.

Для эталонного ТЗ на 30.06 применяется порог 100%.
Для расширенного датасета и корнер-кейсов применяется порог не менее 95%.

Требования со статусом insufficient_detail исключаются из расчета покрытия тест-кейсами, так как они не должны приводить к генерации ТК с додуманными деталями.

## 1. Качество декомпозиции источников на атомарные требования

Критерий

Метод проверки

Метрика / порог

Полнота выделения атомарных требований

Перечень требований, выделенных ЦПТ, сравнивается с эталонным перечнем атомарных требований

Для эталонного ТЗ 30.06 — 100%. Для расширенного датасета — не менее 95%

Корректность границ требований

Проверяется, что разные требования не объединяются в одно, а одно требование не дробится на несколько без необходимости

Границы требований должны соответствовать эталонной разметке. Ошибки границ не засчитываются как корректно выделенные требования

Консолидация требований из разных источников

Проверяется, что информация по одному и тому же требованию, найденная в разных источниках, объединяется, а не создает дубли

Дубли требований не допускаются. Консолидация должна соответствовать эталонной разметке

Автоматическое определение недостаточно детализированных требований

ЦПТ анализирует атомарное требование и определяет, достаточно ли информации для генерации корректных ТК без додумывания отсутствующих деталей

Требования, по которым недостаточно данных для генерации, должны быть автоматически помечены статусом insufficient_detail

Корректность определения статуса insufficient_detail

Статусы, определенные ЦПТ, сравниваются с экспертной разметкой эталонного датасета

Для эталонного ТЗ 30.06 — 100%. Для расширенного датасета — не менее 95%

Трассировка требования до источника

Проверяется, что для каждого атомарного требования сохранена связь с исходным источником, версией источника и фрагментом, из которого требование было выделено

У каждого атомарного требования должна быть сохранена внутренняя связь с источником данных

## 2. Качество генерации тест-кейсов по атомарным требованиям

Критерий

Метод проверки

Метрика / порог

Покрытие атомарных требований тест-кейсами

Для каждого атомарного требования, пригодного для генерации, проверяется наличие связанных с ним сгенерированных ТК

Для эталонного ТЗ 30.06 — 100% пригодных требований должны быть корректно покрыты. Для расширенного датасета — не менее 95%

Полнота покрытия по требованию

Для каждого атомарного требования проверяется наличие позитивного, негативного и иных обязательных проверок из эталона

Требование считается корректно покрытым, если по нему есть необходимый набор ТК согласно эталону

Учет ТК, покрывающих несколько требований

Если один ТК связан с несколькими атомарными требованиями, он учитывается в покрытии каждого связанного требования

Каждый связанный ТК должен быть релевантен каждому требованию, к которому он привязан

Соответствие сгенерированных ТК эталону

Сгенерированные ТК сравниваются с эталонными по полному составу: вид ТК, направление, обязательные проверки, предусловия, шаги, ожидаемые результаты, постусловия, обязательные атрибуты

Сгенерированные ТК должны семантически и структурно соответствовать эталону. Дословное совпадение формулировок не требуется

Допустимость отличий от эталона

Проверяется, что отличия не меняют смысл проверки

Допускаются отличия в формулировках и последовательности шагов, если сохраняется проверяемое требование, покрываются обязательные проверки, не нарушается логика сценария и ожидаемые результаты соответствуют действиям

Дополнительные ТК сверх эталона

Проверяются ТК, которые отсутствуют в эталонном наборе

Дополнительные ТК допустимы только если они релевантны требованию, не дублируют эталонные ТК и не содержат фактов, отсутствующих в требованиях

Корректная классификация вида и направления ТК

Для каждого ТК проверяется, что вид ТК соответствует содержанию шагов и ожидаемых результатов, а направление соответствует характеру входных данных и ожидаемого результата

Вид ТК определяется по расширенной классификации: API, WEB, мобильный, интеграционный, E2E, функциональный, системный. Направление определяется отдельно: позитивный / негативный. Для эталонного ТЗ 30.06 — 100%, для расширенного датасета — не менее 95%

Соответствие шаблону и заполнение обязательных атрибутов

Проверяется, что ТК сформирован по утвержденному шаблону, обязательные атрибуты заполнены, а структура ТК соответствует виду ТК

Обязательные атрибуты должны быть заполнены. Структура ТК должна соответствовать утвержденному шаблону. Содержательное качество заполнения оценивается в рамках соответствия ТК эталону

Качество шагов

Проверяется, что шаги сформулированы как действия, один шаг содержит одно действие, шаги достаточно конкретны и выполнимы

Шаги должны быть однозначными, атомарными и выполнимыми

Качество ожидаемых результатов

Проверяется, что у каждого шага есть ожидаемый результат, он проверяемый, конкретный и логически соответствует действию в шаге

У каждого шага должен быть ожидаемый результат. Ожидаемый результат должен соответствовать шагу и быть проверяемым

Предусловия и постусловия

Проверяется, что предусловия описывают состояние системы до начала сценария, а постусловия не содержат шаги и описывают состояние после выполнения ТК

Предусловия и постусловия должны соответствовать требованию и эталонному ТК, если они применимы

Отсутствие галлюцинаций и необоснованной конкретизации

Проверяется, что ТК не содержит фактов, ролей, UI-элементов, URL, endpoint, текстов ошибок, кодов ответов, ограничений или иных деталей, отсутствующих в требованиях и не следующих однозначно из контекста

Галлюцинации и необоснованная конкретизация не допускаются. ТК с таким нарушением не засчитывается как корректное покрытие атомарного требования

Обработка требований insufficient_detail

Проверяется, что по требованиям с недостаточной детализацией ЦПТ не генерирует ТК с додуманными деталями

Требования insufficient_detail исключаются из расчета покрытия ТК. По ним не требуется генерировать ТК

## 3. Внутренняя трассировка и metadata

ЦПТ должен сохранять внутреннюю трассировку генерации:

источник → атомарное требование → статус требования → сгенерированные ТК

Отображение атомарных требований пользователю в интерфейсе на текущем этапе не требуется.

При этом должна быть предусмотрена служебная ручка / API для получения:

перечня атомарных требований, выделенных в рамках конкретного запуска генерации;

статусов атомарных требований;

связей между атомарными требованиями и сгенерированными ТК.

Минимальный состав metadata по атомарному требованию:

Поле

Назначение

generationRunId

ID запуска генерации

atomicRequirementId

Внутренний ID атомарного требования

requirementText

Текст атомарного требования

requirementStatus

Статус требования: ready_for_generation, covered, partially_covered, insufficient_detail, not_covered, excluded

sourceType

Тип источника: Документы, Знания, Задачи, Реестр API

sourceId

ID исходного объекта

sourceVersion / updatedAt / hash

Версия источника, по которой выполнялась генерация

sourceFragmentRef

Ссылка на раздел, страницу, фрагмент или позицию в источнике

linkedTestCaseIds

Список ТК, связанных с атомарным требованием

Минимальный состав metadata по тест-кейсу:

Поле

Назначение

testCaseId

ID сгенерированного ТК

generationRunId

ID запуска генерации

linkedRequirementIds

Список атомарных требований, которые покрывает ТК

testCaseType

Вид ТК: API, WEB, мобильный, интеграционный, E2E, функциональный, системный

direction

Направление: позитивный / негативный

promptVersion

Версия промпта / шаблона генерации

modelVersion

Версия модели / конфигурации LLM

Связь между ТК и требованиями допускается как многие-ко-многим:

один ТК может покрывать несколько атомарных требований;

одно атомарное требование может покрываться несколькими ТК.

## 4. Расчет покрытия

В расчет покрытия ТК включаются только требования, пригодные для генерации.

Из знаменателя исключаются:

требования со статусом insufficient_detail;

требования, исключенные из генерации по согласованным правилам;

требования, не относящиеся к проверяемому скоупу.

Формула:

Покрытие ТК = количество корректно покрытых требований / количество требований, пригодных для генерации

Пример:

Всего выделено атомарных требований: 100
Недостаточно детализированы: 10
Исключены из генерации: 5
Пригодны для генерации: 85

Корректно покрыто ТК: 81
Покрытие = 81 / 85 = 95,3%
